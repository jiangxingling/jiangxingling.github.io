Android之屏幕适配

发表于 2017-04-19

基本概念

屏幕尺寸、屏幕分辨率、屏幕像素密度

  • 屏幕尺寸

    指屏幕的对角线长度，单位是英寸，1英寸约等于2.54cm

  • 屏幕分辨率

    指在横纵向上的像素点数，单位是px，1px = 1个像素点
    一般是纵向像素 横向像素，如1920 1080

  • 屏幕像素密度

    指每英寸上的像素点数，单位是dpi (dot per inch)
    屏幕像素密度与屏幕尺寸和屏幕分辨率有关
    计算方式（以4.9英寸，19201080的手机为例）：dpi = √(19201920+1080*1080) / 4.9

dp、dip、dpi、sp、px

  • dp、dip

    Density Independent Pixels的缩写，即密度无关像素。
    以160dpi为基准，1dip = 1px，在320dpi上，1dip = 2px
    使用dp/dip可以保证在不同设备上显示相同的效果

  • dpi

    即屏幕像素密度

  • sp

    Scale-Independent Pixels
    常用来设置文字大小

  • px

    构成图像的最小单位。如获取到的屏幕宽高是以px为单位的

mdpi、hdpi、xdpi、xxdpi

  • 对应关系

    mdpi – 120dpi ~ 160dpi
    hdpi – 160dpi ~ 240dpi
    xdpi – 240dpi ~ 320dpi
    xxdpi – 320dpi ~ 480dpi
    xxxdpi – 480dpi ~ 640dpi

解决方案

支持各种屏幕尺寸

  • 使用 wrap_content、 match_parent、 weight
  • 使用相对布局，禁用绝对布局

  • 使用限定符（用于适配平板设备）

    使用最小宽度限定符: 在res下新建layout-large(3.2之前)/layout-sw600dp(3.2之后)文件夹
    使用布局别名: main.xml / main_twopanes.xml
    例：

    默认布局
    res/values/layout.xml:


    @layout/main

    Android 3.2 之前的平板布局
    res/values-large/layout.xml:

    @layout/main_twopanes

    Android 3.2 之后的平板布局
    res/values-sw600dp/layout.xml:

    @layout/main_twopanes

    > 使用屏幕方向限定符
    > 例：
    res/values-sw600dp-land/layout.xml:

    @layout/main_twopanes

    res/values-sw600dp-port/layout.xml:

    @layout/main

Android之内存优化

发表于 2017-04-17

为什么要进行内存优化

在实际开发中，不再用到的对象因为被错误引用导致无法回收，从而造成内存泄漏，甚至内存溢出OOM(OutOfMemory)，程序崩溃。

查看手机内存限制

  1. 获取系统服务和内存限制

  2. 将读取的数据放入StringBuilder中并打印出来

Android 系统内存管理方式

系统内存分配与回收方式

  • 每个App都相当于一个进程对应一个虚拟机

  • 使用GC进行回收

    GC是Java 垃圾处理器的简称，它只在Heap剩余空间不够时才发起垃圾回收动作。当GC被触发时，所有的线程都会被暂停。

App内存限制机制

  • 每个App分配的最大内存限制，根据设备不同而不同
  • 占用内存最厉害的是图片

切换应用时后台App清理机制

  • App 切换时依据LRU Cache
    LRU算法：最近使用的排在最前面，最少可能的被清理掉
  • onTrimMemory()回调方法
    系统内存发生变化时，会调用该方法

监控内存的几种方法

  • 代码监控系统分配给App的内存以及剩余内存

  • 通过Android Monitor监控

  • Tools – Android – Android Device Monitor
    选择要查看的进程，点击Update Heap图标，再点击Cause GC

APP内存优化方法

数据结构优化

  • 频繁字符串拼接用StringBuilder

    通过+的方式进行字符串拼接，会产生中间字符串块，这些都是没有用的

  • 使用ArrayMap、SparseArray替换HashMap
  • 内存抖动
  • 再小的Class也要耗费0.5KB
  • HashMap一个entry需要额外占用32B

对象复用

  • 复用系统自带的资源
  • ListView/GridView的ConvertView复用
  • 避免在onDraw()方法里面执行对象的创建

避免内存泄露

内存泄露：由于代码瑕疵，导致这块内存，虽然已经停止不用了，但依然被其它东西引用着，使得GC没法对它回收。
内存泄露会导致剩余可用的Heap越来越少，频繁触发GC

  • 要注意Activity泄漏
  • 用Application Context而不是Activity Context
  • 注意Cursor对象是否及时关闭

OOM问题优化

强引用与软引用

强引用在其生命周期中不会被GC回收，软引用可以被回收。

优化OOM问题的方法

  • 注意临时Bitmap对象的及时回收
  • 避免Bitmao的浪费
  • Try catch某些大内存分配的操作
  • 加载Bitmap: 缩放比例、解码格式、局部加载

准备工作

  1. Git Bash
  2. github账号
  3. node.js
    到node官网上下载安装即可

安装Hexo

  1. npm 默认的源的下载速度可能很慢，所以使用淘宝镜像替换。执行下面的命令，将npm的源设置成淘宝镜像站：

    npm config set registry “https://registry.npm.taobao.org“

  2. 执行如下命令，安装hexo

    npm install -g hexo-cli

Hexo本地设置

  1. 在E盘下新建Blog文件夹，初始化Hexo

    cd Blog
    hexo init

  2. 安装 npm

    npm install

  3. 开启hexo服务器

    hexo s (或 hexo server)

  4. 打开http://localhost:4000/,可以访问hexo页面，则本地已配置完成
